## Codex Engineering Workflow

本项目使用 mattpocock/skills + ponytail。

### 默认流程

- 需求不清楚时，先使用 `$grill-with-docs`，不要直接写代码。
- 新功能使用 `$tdd`，按最小垂直切片实现。
- 修 Bug 使用 `$diagnosing-bugs`，先复现、再假设、再验证、再修复。
- 大范围改动前使用 `$improve-codebase-architecture`，先输出报告，不直接重构。
- 长任务结束后使用 `$handoff` 生成交接文档。

### Ponytail 规则

- 默认使用 `@ponytail full`。
- 优先删除代码，不新增抽象。
- 优先使用已有代码、已有依赖、标准库、平台能力。
- 禁止为了“以后可能需要”新增配置、接口、工厂、抽象层。
- 禁止新增无必要依赖。
- 禁止新增只有一个实现的 interface / adapter / factory。
- 非复杂逻辑不要引入大型测试框架。
- 非平凡逻辑必须留下一个最小可运行检查。
- 对有意简化但未来可能扩展的地方，用 `ponytail:` 注释说明上限和升级路径。

### 项目约束

- 修改 Python 后端后必须运行相关 pytest，或提供 curl 验证命令。
- 修改 Qt/QML 后必须说明受影响页面、接口和构建命令。
- 不允许改变现有 Dashboard 视觉风格，除非需求明确要求。
- 不允许把真实数据链路退回 mock。


# 公共工程约束（Architecture First）

## 总体原则
- 先设计后编码，分析架构、职责、调用关系，避免重复实现。
- 优先最小改动，保持兼容。

## 软件工程原则
- 遵循 SOLID、KISS、DRY、YAGNI、Fail Fast。
- 高内聚、低耦合、模块化。

## 单一职责
- 一个文件/类/函数只负责一项职责。
- pipeline 只编排，不实现业务。

## 设计模式
- 按需使用 Strategy、Factory、Facade、Observer、DI 等，不为模式而模式。

## 代码风格
- 全项目统一命名、日志、异常、类型注解、Docstring、Import 顺序。
- 文件建议<=500行；函数建议<=80行。

## 配置
- API、路径、超时、线程数统一配置，禁止硬编码。

## 日志与异常
- 禁止 print、except: pass。
- 使用 logging，记录上下文。

## 同步/异步
- CPU密集→ProcessPool；IO密集→AsyncIO/ThreadPool。
- 合理并发，按依赖编排任务。

## 任务拆分
- P0/P1/P2/P3 优先级。
- 每次提交保持单一主题，可测试、可回滚。

## API保护
- 实现限流、最大并发、指数退避、重试、熔断。
- 防止429和服务雪崩。

## 资源管理
- 使用 with/context manager。
- 正确释放文件、连接、线程池。

## 测试
- 新功能提供最小验证。
- 保证可测试，减少网络/文件系统耦合。

## 文档
- 修改接口、配置、CLI后同步 README、CHANGELOG。

## AI修改原则
- 优先复用现有实现。
- 禁止无必要重构。
- 保持接口兼容。
